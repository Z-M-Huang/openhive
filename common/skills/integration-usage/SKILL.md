## When to Apply

Use this skill when your task involves interacting with an external service -- sending notifications, fetching data from APIs, processing webhooks, or connecting to third-party platforms. Integrations are the bridge between OpenHive agents and the outside world.

## Integration Lifecycle

Integrations follow a three-phase lifecycle: create, test, activate.

```
Discover need --> Create integration config --> Test connection --> Activate
                       |                            |
                       v                            v
                  Set credentials              Fix issues
                  (if needed)                  and re-test
```

## Steps

### 1. Check for Existing Integrations

Before creating a new integration, check if one already exists:

- Look in `/app/workspace/integrations/` for YAML config files.
- Check your memory for notes about previously configured integrations.
- Ask your team lead if unsure whether an integration exists at a higher level.

### 2. Configure Credentials

Most integrations require credentials. Use the credential tools to manage them securely:

```
set_credential:
  key: "GITHUB_TOKEN"
  value: <token-value>
  scope: "team"  # available to all agents in this team
```

```
get_credential:
  key: "SMTP_PASSWORD"
```

Credential rules:
- Never hardcode credentials in integration config files.
- Never include credentials in task results, messages, or logs.
- Use `set_credential` for new credentials received from escalation responses.
- Use `get_credential` to retrieve credentials at invocation time.
- Credentials are encrypted at rest via the KeyManager (AES-256-GCM).

### 3. Create the Integration

Use `create_integration` to define a new integration:

- Provide a clear name that describes the service and purpose.
- Specify the integration type (webhook, API, scheduled, event-driven).
- Include configuration parameters specific to the service.

The integration config is written to `/app/workspace/integrations/<name>.yaml`.

### 4. Test Before Activating

Always test an integration before activating it:

```
test_integration:
  name: "github-pr-notifications"
```

Testing verifies:
- Credentials are valid and have sufficient permissions.
- The external service is reachable.
- The request/response format matches expectations.
- Rate limits and quotas are within bounds.

If the test fails:
- Check credential validity and permissions.
- Verify the service URL and configuration parameters.
- Check network connectivity from the container.
- Review error messages for specific guidance.
- Fix the issue and re-test before proceeding.

### 5. Activate the Integration

Once testing passes, activate the integration:

```
activate_integration:
  name: "github-pr-notifications"
```

Activation makes the integration available for use by agents in this team. The integration begins processing events (for webhook/event-driven types) or becomes callable (for API types).

### 6. Register Webhooks (If Applicable)

For integrations that receive inbound events, register a webhook endpoint:

```
register_webhook:
  integration: "github-pr-notifications"
  events: ["pull_request.opened", "pull_request.merged"]
```

The webhook URL is generated by the system and must be configured in the external service's settings.

## Using Helper Scripts

Common I/O operations are available via helper scripts in `/app/common/scripts/`. Reference these in your integration workflows:

| Script | Purpose |
|--------|---------|
| `/app/common/scripts/send-email.py` | Send email via SMTP with optional attachments |
| `/app/common/scripts/http-client.ts` | Typed HTTP requests with timeout and retry |
| `/app/common/scripts/parse-rss.py` | Parse RSS/Atom feeds into structured data |
| `/app/common/scripts/render-template.ts` | Render Mustache templates from data |
| `/app/common/scripts/format-html-email.ts` | Convert markdown to styled HTML email |

Scripts read credentials from environment variables, not from config files. Ensure the required environment variables are set via `set_credential` before invoking scripts.

## Integration Config Format

Integration configs are YAML files in `/app/workspace/integrations/`:

```yaml
name: github-pr-notifications
type: webhook
description: "Notify team channel when PRs are opened or merged"
service: github
endpoint: "https://api.github.com"
auth:
  type: token
  credential_key: GITHUB_TOKEN    # References stored credential
events:
  - pull_request.opened
  - pull_request.merged
actions:
  on_event:
    task:
      title: "Review PR"
      prompt: "Review the following pull request: {{payload}}"
```

## Troubleshooting

| Symptom | Likely Cause | Action |
|---------|-------------|--------|
| Test returns 401/403 | Invalid or expired credentials | Refresh credential via `set_credential` |
| Test returns timeout | Service unreachable or firewall | Check container network, verify URL |
| Webhook not firing | Webhook URL not registered in external service | Re-register and verify external config |
| Duplicate events | Missing deduplication in event handler | Check integration config for idempotency settings |
